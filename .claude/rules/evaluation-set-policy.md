# Evaluation Set Policy

**テストコードを追加・変更したら、対象パッケージの評価セット（`docs/evaluation/<pkg>.md`）と突合し、ギャップゼロで完了する。**

## 原則

- `docs/evaluation/<pkg>.md` は各パッケージの「あるべき状態」の SSOT である
- テストの妥当性はテスト自体ではなく評価セットで判断する（AI 生成テストの実装追認を防ぐ）
- 評価セットと実装・テストが矛盾する場合は評価セットを正とする。評価セット自体が誤っていると思われる場合は人間に確認する

## テスト改修時の手順（MUST）

テストコード（`packages/<pkg>/tests/` / `tests/unit/` / `tests/e2e/`）を追加・変更・削除したら:

1. 変更したテストが属するパッケージを特定する
2. `docs/evaluation/<pkg>.md` を読み、評価観点（`EV-NN`）を確認する
3. 突合マトリクスを一時生成する: `EV ID | 対応テスト | 優先度 | covered/gap`
4. **変更スコープに関係する `must` 観点がすべて covered であることを確認してから完了する**
5. 新たに見つかったギャップは、パッケージ別ギャップ Issue に追記する（存在しなければチェックリスト形式で新規作成する）
6. マトリクス自体は評価セットや成果物に恒久記録しない（陳腐化するため。ギャップの記録先は Issue のみ）

対応する評価セットが存在しないパッケージのテストを変更する場合は、その旨を報告し、評価セットの新規作成を提案する。

## 仕様変更時の手順

パッケージの振る舞い（仕様）を変更する場合:

1. 先に `docs/evaluation/<pkg>.md` の該当観点を更新する
2. 人間レビューを経て、ヘッダの「最終レビュー日」を更新する
3. テスト・実装を評価セットに追従させる

## AI 生成テストのレビュー基準

`docs/evaluation/README.md` の「テストレビュー判断基準（共通）」6 項目と、各評価セットの「5. テストレビュー判断基準（パッケージ固有）」を適用する。

## 新規パッケージ

新しいパッケージを追加する際は、`docs/evaluation/_template.md` に従い評価セットも同時に作成する。

## スキルフロー評価セット（docs/evaluation/skills/）

facets 由来のスキル群には、フロー単位の評価セット（`docs/evaluation/skills/<flow>.md`）を置く。

- スキルは Markdown 指示書であり pytest で強制できないため、**本ルールの「テスト改修時の手順（MUST）」の対象外**（振る舞い仕様書として扱う）
- スキルフローの仕様（フェーズ構成・ゲート条件・成果物パス等）を変更する PR では、該当する `skills/<flow>.md` と矛盾しないか突合する。仕様変更の場合は評価セットを先に更新し、人間レビューを経る
- 検証手段は各評価セットの「検証方法」（PR レビュー時の突合 / `/config-analyze` / 実行観察）に従う

## 自動化

テストファイル変更時の突合確認は、quality-gates パッケージの `evaluation-set-checker.py` hook（PostToolUse: Edit|Write）が案内する。有効/無効は `audit-flags.json` の `features.evaluation_set_check.enabled` で切替できる（`.local.json` で上書き可。Issue #123）。

hook はあくまで確認を促す案内であり、突合作業自体（マトリクス生成・must 観点のカバレッジ確認・ギャップの Issue 追記）は上記「テスト改修時の手順（MUST）」に従いオーケストレーターが実施する。
