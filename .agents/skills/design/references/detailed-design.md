# 詳細設計フェーズ（Phase 3）詳細手順

このドキュメントは `/design` スキルの Phase 3 で参照される。

## 前提

Phase 2 の基本設計が完了しているか、ユーザーが既にアーキテクチャや設計方針を把握していること。
`docs/` 配下の設計ドキュメントが存在する場合はまず読み込んで文脈を把握する。

## 目的

基本設計の一覧ドキュメントから、各項目を個別の設計書ファイルに展開する。
実装者がこのドキュメントだけで実装を始められるレベルの詳細を記述する。

## 基本方針

- 基本設計の「一覧」に対して、各項目の「個別設計書」を作成する
- 個別設計書のファイル名は一覧の ID に対応させる（`API-001.md`, `SC-001.md` 等）
- 一度に全項目を詳細化する必要はない — 優先度の高い項目（Must）から段階的に進める
- ユーザーと対話しながら、各設計書の内容を確定していく

## ステップ

基本設計で作成した一覧ドキュメントに対応するステップを実施する。

### Step 1: API 詳細設計

`docs/api/api-list.md` の各エンドポイントに対して個別設計書を作成する。

#### 出力

各エンドポイントごとに `docs/api/API-{ID}.md` を作成する。

#### 対話の進め方

1. `api-list.md` を読み込み、エンドポイント一覧を確認
2. 優先度の高いエンドポイントから順に対話で詳細化
3. エンドポイントごとに「リクエストには何が必要ですか？」「エラー時はどうしますか？」と具体化
4. 仕様をまとめてユーザーに提示し、合意を得てからファイル化

#### 個別設計書のフォーマット（API-{ID}.md）

````markdown
# API-001: {エンドポイント名}

## 概要

- **メソッド**: POST
- **パス**: /api/v1/reviews
- **概要**: {説明}
- **認証**: 要
- **対応機能**: FT-xxx

## リクエスト

### ヘッダー

| ヘッダー      | 値               | 必須 | 説明             |
| ------------- | ---------------- | ---- | ---------------- |
| Authorization | Bearer {token}   | Yes  | アクセストークン |
| Content-Type  | application/json | Yes  |                  |

### パスパラメータ

| パラメータ | 型   | 説明   |
| ---------- | ---- | ------ |
| {名前}     | {型} | {説明} |

### クエリパラメータ

| パラメータ | 型   | 必須   | デフォルト   | バリデーション | 説明   |
| ---------- | ---- | ------ | ------------ | -------------- | ------ |
| {名前}     | {型} | {必須} | {デフォルト} | {ルール}       | {説明} |

### リクエストボディ

| フィールド | 型            | 必須 | バリデーション | 説明     |
| ---------- | ------------- | ---- | -------------- | -------- |
| product_id | string (UUID) | Yes  | 存在する商品ID | 対象商品 |
| rating     | integer       | Yes  | 1-5            | 星評価   |
| title      | string        | Yes  | 1-100文字      | タイトル |
| body       | string        | No   | 最大2000文字   | 本文     |

## レスポンス

### 201 Created

```json
{
  "id": "uuid",
  "product_id": "uuid",
  "rating": 5,
  "title": "素晴らしい商品",
  "body": "...",
  "created_at": "2026-01-01T00:00:00Z"
}
```
````

| フィールド | 型                | 説明       |
| ---------- | ----------------- | ---------- |
| id         | string (UUID)     | レビューID |
| product_id | string (UUID)     | 商品ID     |
| rating     | integer           | 星評価     |
| title      | string            | タイトル   |
| body       | string            | null       | 本文 |
| created_at | string (ISO 8601) | 作成日時   |

### 400 Bad Request

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容に誤りがあります",
    "details": [
      { "field": "rating", "message": "1から5の間で指定してください" }
    ]
  }
}
```

### 401 Unauthorized

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "認証が必要です"
  }
}
```

## ビジネスルール

- {ビジネスルール1}
- {ビジネスルール2}

## 備考

- {補足事項}

````

### Step 2: 画面詳細設計

`docs/screens/screen-list.md` の各画面に対して個別設計書を作成する。

#### 出力

各画面ごとに `docs/screens/SC-{ID}.md` を作成する。

#### 対話の進め方

1. `screen-list.md` を読み込み、画面一覧を確認
2. 優先度の高い画面・主要フローの画面から順に対話で詳細化
3. 「この画面にはどんな要素が必要ですか？」と要素を洗い出す
4. 「入力に対してどんなバリデーションが必要ですか？」と制約を確認
5. エラー時やデータがない場合の表示を確認

#### 個別設計書のフォーマット（SC-{ID}.md）

```markdown
# SC-001: {画面名}

## 概要

- **画面名**: {名前}
- **概要**: {説明}
- **対応機能**: FT-xxx
- **遷移元**: {画面名}
- **遷移先**: {画面名}

## 構成要素

| 要素 | 種類 | 説明 | 必須 |
|------|------|------|------|
| 星評価 | レーティング入力 | 1-5の星を選択 | Yes |
| タイトル | テキスト入力 | 100文字以内 | Yes |
| 本文 | テキストエリア | 2000文字以内 | No |
| 投稿ボタン | ボタン | バリデーション通過後に活性化 | - |

## バリデーション

| フィールド | ルール | エラーメッセージ |
|-----------|--------|----------------|
| 星評価 | 未選択不可 | 「評価を選択してください」 |
| タイトル | 空文字不可、100文字以内 | 「タイトルを入力してください」 |

## 状態管理

| 状態 | 表示 | トリガー |
|------|------|---------|
| 初期表示 | 空のフォーム | 画面遷移時 |
| 入力中 | 入力内容を表示 | ユーザー入力 |
| 送信中 | ボタン無効化 + スピナー | 投稿ボタン押下 |
| 成功 | 完了メッセージ | API成功レスポンス |
| エラー | エラーメッセージ表示 | APIエラーレスポンス |
| 空状態 | {データがない場合の表示} | データ0件 |

## API 連携

| アクション | API | タイミング |
|-----------|-----|-----------|
| {操作} | API-001 | {タイミング} |

## プラットフォーム固有の考慮

{Web / iOS / Android で異なる挙動がある場合に記載}

## 備考

- {補足事項}
````

### Step 3: データモデル詳細設計

`docs/database/er-design.md` のテーブル定義について、必要に応じて個別ファイルに展開する。

基本設計の `er-design.md` に既にテーブル定義（カラム、型、制約）が含まれているため、Phase 3 で個別ファイルに分離するのは以下の場合：

- テーブル数が多く `er-design.md` が長大になった場合
- テーブルごとに複雑なビジネスルールやマイグレーション戦略がある場合
- 個別テーブルの設計を深く議論する必要がある場合

#### 出力（必要な場合のみ）

`docs/database/TBL-{ID}.md` として個別テーブル設計書を作成する。

#### 個別設計書のフォーマット（TBL-{ID}.md）

```markdown
# TBL-001: {テーブル名}

## 概要

{テーブルの目的と役割}

## カラム定義

| カラム     | 型          | 制約     | デフォルト        | 説明     |
| ---------- | ----------- | -------- | ----------------- | -------- |
| id         | UUID        | PK       | gen_random_uuid() | 主キー   |
| {名前}     | {型}        | {制約}   | {デフォルト}      | {説明}   |
| created_at | TIMESTAMPTZ | NOT NULL | NOW()             | 作成日時 |
| updated_at | TIMESTAMPTZ | NOT NULL | NOW()             | 更新日時 |

## インデックス

| 名前   | カラム   | 種類   | 目的   |
| ------ | -------- | ------ | ------ |
| {名前} | {カラム} | B-tree | {目的} |

## リレーション

| 関係先     | 種類 | FK     | ON DELETE |
| ---------- | ---- | ------ | --------- |
| {テーブル} | N:1  | {FK名} | CASCADE   |

## ビジネスルール

- {このテーブルに関連するビジネスルール}

## マイグレーション

{既存システムからの移行がある場合の戦略}
```

### Step 4: コンポーネント詳細設計

アーキテクチャで定義した主要コンポーネントの内部構造を設計する。

#### 設計内容

- モジュール構成（ディレクトリ構造）
- 各モジュールの責務とインターフェース
- モジュール間の依存関係
- 共通ユーティリティ

#### 対話の進め方

1. `docs/architecture/architecture.md` のコンポーネント一覧を確認
2. 「このコンポーネントの内部はどういう構造にしますか？」
3. 「外部とのインターフェースはどうしますか？」
4. ディレクトリ構造案を提示してフィードバックをもらう

#### 出力

`docs/architecture/` 配下に必要に応じてコンポーネント別ファイルを追加する。

## ステップ選択の判断基準

| 基本設計の成果物                | 対応する詳細設計ステップ               |
| ------------------------------- | -------------------------------------- |
| `api-list.md` あり              | Step 1（API 詳細） → `API-{ID}.md`     |
| `screen-list.md` あり           | Step 2（画面詳細） → `SC-{ID}.md`      |
| `er-design.md` のテーブルが複雑 | Step 3（テーブル詳細） → `TBL-{ID}.md` |
| アーキテクチャが複雑            | Step 4（コンポーネント詳細）           |

基本設計で作成したドキュメントに対応するステップのみ実施する。

## 注意事項

- 詳細設計は「実装者がこのドキュメントだけで実装を始められるか？」を基準にする
- 過度に細かくしすぎない — 実装時に判断できる程度の裁量は残す
- 基本設計との整合性を保つ — 詳細設計で方針変更が必要になった場合は基本設計側も更新する
- 全項目を一度にやる必要はない — 優先度の高い項目（Must）から段階的に詳細化する
- 個別設計書のIDは一覧の ID と必ず対応させる
- `docs/glossary.md` の用語を一貫して使用する

## セルフチェックリスト（品質ゲート Stage 1）

成果物の作成が終わったら、レビュアー起動・受け入れ確認の前にオーケストレーター自身が以下を検査する。
未達項目があれば成果物を修正してから次に進む。

### 完成基準

- [ ] `api-list.md` に記載された全エンドポイント（少なくとも Must 優先度分）に対応する `API-{ID}.md` が存在する
- [ ] `screen-list.md` に記載された全画面（少なくとも Must 優先度分）に対応する `SC-{ID}.md` が存在する
- [ ] 各 `API-{ID}.md` にリクエスト（パラメータ/ボディ）とレスポンス（成功時 + 主要エラー: 400/401等）の両方が定義されている
- [ ] 各 `API-{ID}.md` のリクエストフィールドにバリデーションルールが明記されている（型のみで欄が空でない）
- [ ] 各 `SC-{ID}.md` に構成要素・バリデーション・状態管理（初期表示/エラー/空状態等）が定義されている
- [ ] 各 `SC-{ID}.md` の「API 連携」欄に記載された API が `api-list.md` または個別 `API-{ID}.md` に実在する
- [ ] 個別設計書のファイル名・見出しの ID が一覧側の ID と一致している（`API-001.md` の見出しが `API-001` である等）
- [ ] 基本設計（`architecture.md`, `er-design.md` 等）の内容と矛盾する記述がない
- [ ] このドキュメントだけで実装を開始できるかという基準で、未記載の共通仕様への暗黙の依存がない

### アンチパターン（該当したら要修正）

- **正常系のみ設計**: エラーレスポンス（400/401/403/404等）が個別設計書に記載されていない → `api-list.md` の共通エラーフォーマットを踏まえて追記する
- **暗黙の共通仕様依存**: 「共通仕様に従う」とだけ書いて具体値がどこにもない → 該当する具体的な値・挙動を明記する
- **一覧との ID 不一致**: 個別ファイル名やタイトルの ID が一覧（`api-list.md`/`screen-list.md`）の ID とずれている → ID を揃える
- **過剰な詳細化**: 実装時に決めればよい内部実装詳細（変数名等）まで規定している → 実装者の裁量を残すレベルに戻す
- **基本設計との齟齬**: 詳細設計で方針変更が発生したのに基本設計側の一覧・方針が更新されていない → 基本設計側も同期して更新する
