# ADR-0014: 自動承認を安全判定 reviewer 経由にする

## Status

Accepted

## Date

2026-07-07

## Context

baton の自動承認モードは、セッション単位で有効にすると `Waiting` 検出時に常に `Enter` を送信していた。

この方式は承認疲れを減らせる一方で、AI エージェントが危険な操作を提案した場合にも、そのまま承認してしまう。
特に以下の操作では、人間の判断へ戻す必要がある。

- 破壊的なファイル操作（`rm` など）
- Git 履歴や未保存作業に影響する操作（`git reset` / `git clean` など）
- 秘密情報・認証情報へのアクセス
- workspace 外への書き込み
- 外部ネットワーク送信やデータ持ち出し
- production / deploy / infrastructure 操作

Claude Code の auto mode や Codex Auto-review は、既存の sandbox / permission 境界を緩めず、承認直前に別の classifier / reviewer を挟む設計を採用している。
baton でも同じ考え方を採用し、無条件承認ではなく「安全と判断できる場合だけ承認」する必要がある。

## Decision

自動承認は、以下の順序で安全判定してから実行する。

1. ルールベースの deterministic guard で明確に低リスクな操作と危険操作を先に分類する
2. ルールだけでは曖昧な操作は Codex exec reviewer に渡す
3. reviewer の初期モデルは `gpt-5.3-codex-spark` とする
4. `allow` かつ risk が設定閾値以下の場合のみ `Enter` を送信する
5. `ask` / `deny` / `unknown` / `error` はキー送信せず、TUI に停止理由を表示して人間の操作を待つ

`deny` であっても `Escape` は自動送信しない。
baton は対象セッションを勝手に拒否して会話を進めるのではなく、ユーザーに判断材料を返す。

設定は `auto_mode` に集約し、reviewer 種別、モデル、タイムアウト、リスク閾値を変更可能にする。

## Rationale

- **安全境界を緩めない**: sandbox や各AIツールの権限設定は変更せず、baton の自動キー送信前に判定を追加するだけに留める
- **低リスク操作は高速に通す**: `go test` など明確に低リスクなローカル操作は LLM 呼び出しなしで承認できる
- **曖昧な操作は保守的に止める**: reviewer 失敗・タイムアウト・JSON不正・不明リスクはすべて停止扱いにする
- **ユーザーの最終判断権を残す**: 危険・不明時は自動拒否せず、既存の `a` / `d` / `A` / `D` で手動判断できる
- **provider 差し替え余地を残す**: MVP では Codex Spark を使うが、reviewer interface により将来 OpenCodeGo などを追加できる

比較した代替案:

- **従来どおり無条件 Enter**: 承認疲れは最小だが、危険操作も承認するため不採用
- **すべて LLM 判定に送る**: 設計は単純だが、低リスク操作にも遅延とコストが発生するため不採用
- **危険判定時に自動で拒否キーを送る**: 会話の進行を baton が決めすぎるため不採用
- **完全なポリシー DSL を先に作る**: 将来性はあるが MVP として大きすぎるため後回し

## Consequences

### Positive

- 自動承認の利便性を維持しつつ、危険操作の無条件承認を避けられる
- 失敗時は停止に倒すため、LLM reviewer の可用性や精度に過度に依存しない
- 停止理由が TUI に出るため、ユーザーが手動判断しやすい
- 既存の手動承認/拒否操作を維持できる

### Negative

- 曖昧な承認では Codex exec reviewer の起動時間と利用コストが増える
- ペインテキストから承認対象を抽出するため、各AIツールのUI変更に影響される
- ルールと reviewer の両方を保守する必要がある
- reviewer が保守的に判断すると、従来より停止回数が増える
