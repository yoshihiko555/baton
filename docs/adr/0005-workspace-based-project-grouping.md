# ADR-0005: ワークスペースベースのプロジェクトグルーピング

## Status

Accepted

## Date

2026-03-14

## Context

WezTerm の運用をワークスペース駆動（プロジェクト別ワークスペース + 複数ワークスペース常時起動）に移行した。baton v2 の設計では `wezterm cli list` の `workspace` フィールドを無視しており、CWD のみでプロジェクトをグルーピングしていた。

全ワークスペースを横断監視し、ワークスペース名をプロジェクトの表示名として使いたいというニーズが発生した。

## Decision

`Pane` 構造体に `Workspace string` フィールドを追加し、プロジェクトのグルーピングキーをワークスペース優先に変更する。

- Workspace が空でなく `"default"` でもない場合 → Workspace でグルーピング、ワークスペース名を `Project.Name` に使用
- Workspace が空 or `"default"` の場合 → CWD でグルーピング（既存の動作を維持）

## Rationale

- **1:1 対応の前提**: 1 ワークスペース = 1 プロジェクトの運用であり、ワークスペース名がプロジェクトの自然な識別子になる
- **`"default"` のフォールバック**: WezTerm は初期状態で全 pane を `"default"` に属させるため、ワークスペースを使わないユーザーへの影響を排除できる
- **表示名の改善**: CWD ベース名（パスの末尾）よりもワークスペース名の方が人間に読みやすい
- **データは既に取得可能**: `wezterm cli list --format json` は全ワークスペースの pane を返しており、`workspace` フィールドを無視していただけ

比較した代替案:
- **CWD のみを維持**: ワークスペース駆動運用に対応できない
- **ワークスペースのみでグルーピング**: `"default"` ワークスペースを使うユーザーで全 pane が1プロジェクトにまとまってしまう

## Consequences

### Positive

- ワークスペース駆動運用でプロジェクトが自然にグルーピングされる
- プロジェクト表示名がワークスペース名になり、視認性が向上する
- ワークスペースを使わないユーザーの動作に影響しない（`"default"` フォールバック）
- JSON 出力に `workspace` フィールドが追加され、外部ツールからの利用性が向上する

### Negative

- 1 ワークスペース内に複数 CWD がある場合、全て同一プロジェクトにグルーピングされる（1:1 前提の運用制約）
- `projectKey` 型の導入により、グルーピングロジックがやや複雑になる
